# # TRACK の DTL コード理解メモ

## 目的

- TRACK の DTL拡張について、まずはコード構成と実行時の入出力関係を理解する。
- その後 ImpactX への移植方法を考える。

---

## DTL の基本動作


1. 外部から RF 電力を空洞へ入れ、加速ギャップに軸方向電場を立てる。
    
2. 粒子はdrift tube 間のギャップ部で加速される。
    
3. 減速位相の間は drift tube 内を通過するようにセル長を設計し、減速を避ける。
    
4. このままでは横方向に発散しうるため、横方向集束として四極磁場をかける
    

---

## ファイル構成

TRACK の DTL 実行に関係する主要ファイルは、以下。

- `track.dat`
- `sclinac.dat`
- `DTL.#nn`
- `eh_DTL.#nn`
- `eh_SOL.#nn`
    

`track.dat` は通常のグローバル設定、`sclinac.dat` はラティス定義、`DTL.#nn` は DTL 要素ごとの補助設定、`eh_DTL.#nn` は DTL セルの電場マップ、`eh_SOL.#nn` はソレノイド磁場マップ。

- 電場は DTL セル用マップ 
- 磁場は別ファイルのソレノイドマップ

DTLのデフォルト動作では、DTLのマップには四極磁場を含むはずだが、見当たらない。。
四極磁場集束ではなく、**ソレノイド集束を使っている構成**しているのでは？


---

## `sclinac.dat` の役割

`sclinac.dat` では、DTL 要素が次の形式で与えられる。

```text
nn  DTL  len  rad  vscale  harm  phase  ncells  nstep
```

ここで各パラメータは概ね以下を意味する。

- `nn`: DTL 要素番号
- `len`: DTL 長さ
- `rad`: 半径
- `vscale`: 電場スケール因子
- `harm`: 高調波数
- `phase`: 位相
- `ncells`: セル数
- `nstep`: 積分ステップ数
    
現状の入力では、`nn = 1 ... 59` の DTL 要素が並んでおり、それぞれに対して `vscale` と `phase` が変化している。したがって、少なくとも `sclinac.dat` 側では、**各 DTL 要素ごとに RF 振幅と位相を調整している**と理解できる。

---

## `DTL.#nn` の役割

`DTL.#nn` には、少なくとも以下の項目が入る。

```
Cell# Length Radius Eflvl E0 phase Ql-len Qr-len Ql-Bgrd Qr-Bgrd Ectr-Gctr
```

- `Cell #`
- `Length`
- `Radius`
- `Eflvl`
- `E0`
- `phase`
- `Ql-len`
- `Qr-len`
- `Ql-Bgrd`
- `Qr-Bgrd`
- `Ectr-Gctr`
    

現状確認できた `DTL.#01`〜`DTL.#99` では、概ね以下が固定されている。

- `Length = 30 cm`
- `Radius = 10 cm`
- `Eflvl = 1`
- `E0 = 1 MV/m`
- `phase = -45 deg`
- `Ql-len = 30 cm`
- `Qr-len = 30 cm`
- `Ectr-Gctr = 0 cm`
    

一方で、変化しているのは主として

- `Ql-Bgrd`
- `Qr-Bgrd`
    

である。

- ただし、現時点ではこの `Ql-Bgrd`, `Qr-Bgrd` が実際に計算で効いているかは不明。

	1.  **名残としてパラメータだけ残り、現実装では未使用の可能性？**
	2.  ソレノイドの強度を変更する可能性？

---

## `eh_DTL.#nn` の役割

- `eh_DTL.#nn` は DTL セル用の場ファイル
- manual の説明と完全には一致しない。

manual では、`eh_DTL.#nn` に

- 3D 磁場マップ    
- 各セルの 2D 電場マップ
    
をまとめて持つ形式が想定されている。  
しかし、手元のファイルでは先頭に

```text
SOL : 64 -> Read eh_SOL.#nn
Cell : 1 -> Read following cell field data
```

のようなメタ情報があり、その後ろにセル電場データらしき数表が続いている。  
したがって、手元環境では

- `eh_DTL.#nn` は **DTL セル電場マップ**
- 磁場は `eh_SOL.#64` から別途参照
    
という実装になっていそう。

- また、`eh_DTL.#01`〜`eh_DTL.#99` は md5 が同一であり、内容も共通。
- 差があるとすれば、それは `sclinac.dat` 側の `vscale`, `phase`、および `DTL.#nn` 側の付加パラメータである。

### `eh_DTL.#nn` の読み込み対応

- `eh_DTL.#nn` の電場側の読み込み形式は、manual の `Read E field sequentially cell by cell` の記載と**概ね一致**している。
-  **`ksym = 2` と解釈してよさそう**。
    
    - `0. 10. 25` は $r_{\min}=0,\quad r_{\max}=10,\quad n_r=25$
    - `0. 15. 26` は $z_{\min}=0,\quad z_{\max}=15,\quad n_z=26$ 
      に対応していると読める。
        
- `ksym = 2` の場合、manual では **z 方向後半のみを与え、前半を対称補完**する。
 - したがって内部的には  
   $$
    z \in [0,30],\quad N_z = 2\times 26 - 1 = 51  
    $$
    の全セル電場に復元される。
    
- これは `DTL.#nn` に書かれている  $$
    \text{Length} = 30  
    $$ と整合する。
    

---

## `eh_SOL.#64` の役割

`eh_SOL.#64` のヘッダ部分は

```text
1
0 0 0
-10 10 25
-10 10 25
-15 15 201
```

eh_SOL.#64 冒頭は、manual の B-field 読み込み部とそこそこ対応。

- 1行目: `icell`（ただし `d_cell`, `aperture` は無し） (セル数1なのであっている)
- 2行目: `ksym_x, ksym_y, ksym_z = 0,0,0` (対称性はなさそうなデータなのであっている)
- 3行目: `xmini, xmaxi, nxx = -10, 10, 25` (アパーチャ 10 cm ともあっている)
- 4行目: `ymini, ymaxi, nyy = -10, 10, 25`
- 5行目: `zmini, zmaxi, nzz = -15, 15, 201` (長さ30cmともあっている)

従って、`eh_SOL.#64` は `[-10,10] x [-10,10] x [-15,15]` の `25 x 25 x 201` 格子を持つ xyz-3D 磁場マップと見てよい その後ろは `bx`, `by`, `bz` の配列本体に対応するはずだが、Fortran の free-format read なので、表示上の各行 3 列をそのまま `(Bx, By, Bz)` と解釈とすべきか。

> [!note]
> manual場は、bx を連続 N点読み出し　→　by を。。。
> となっているが、eh_SOL.#nnは、bx, by, bz で定義されていそう（bx~byのため）



## `nn` と `Cell #` の関係

`DTL.#nn` のファイル名に付く `nn` は、ラティス中の DTL 要素番号である。  
一方、`DTL.#nn` の中に書かれている `Cell #` は、本来はセル番号として解釈されるべき別概念である。

ただし、現状の入力では `sclinac.dat` の各 DTL 要素が実質的に **単セル DTL** として使われているように見える。  
そのため、運用上

- `DTL.#01` の中に `Cell # = 1`
    
- `DTL.#02` の中に `Cell # = 2`
    

のように、ファイル番号とセル番号を一致させているだけと考えられる。  
したがって、**本来は別概念だが、この入力では 1 要素 = 1 セルとして運用されているため、同じ番号になっている**と理解する。

---

## 現状の最も自然な実行像

以上を総合すると、現状の TRACK DTL 実行は概ね以下のように整理できる。

- `sclinac.dat` で各 DTL 要素の `vscale`, `phase` を指定する。
    
- `DTL.#nn` で各要素に対するセル付加情報を読む。
    
- `eh_DTL.#nn` からセル電場マップを読む。
    
- `eh_SOL.#64` からソレノイド磁場マップを読む。
    
- 位置ごとの `E`, `B` を用いてローレンツ力で粒子を追跡する。
    

図式化すると、

```text
sclinac.dat
  ├─ DTL 番号 nn
  ├─ vscale, phase
  ↓
DTL.#nn
  ├─ セル付加情報
  ↓
eh_DTL.#nn
  ├─ 電場マップ  E = (Er, Ez)
  ↓
eh_SOL.#64
  ├─ 磁場マップ  B = (Bx, By, Bz)
```

という理解になる。

---

## 現段階での判断

現時点では、以下の理解を暫定的な結論とする。

1. `eh_DTL.#nn` は電場マップである。    
2. `eh_SOL.#64` はソレノイド磁場マップである。
3. `DTL.#nn` の `Ql-Bgrd`, `Qr-Bgrd` が有効かは不明（ソレノイドの強度変更用とみる）
4. 現実装では **四極磁場ではなく、ソレノイド磁場による集束**が使われているとみる

---

## 今後の方針

ソースコードが無く、未確定事項については**パラメータ感度試験で切り分ける**。

確認すべき点は以下。

- 電場マップと磁場マップのプロットを作成してマップを確認。
- `Ql-Bgrd`, `Qr-Bgrd` を変えたときに結果が変わるか。
- `eh_SOL.#64` を変更したときに横方向集束が変わるか。
- `vscale`, `phase` を変更したときにエネルギー・位相空間が変化するか

この切り分けにより、
- 四極パラメータが実際に効いているか
- 実質的な磁場ハンドルが `eh_SOL.#64` でよいか。
を判断する。
